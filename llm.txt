# md-pdf — LLM reference

> This file is designed for large language models assisting users with md-pdf.
> Canonical URL: https://raw.githubusercontent.com/Moravio/homebrew-md-pdf/main/llm.txt
> Version: 3.20.1

md-pdf is a CLI + library that converts Markdown files into professionally
branded PDFs with title pages, table of contents, Mermaid diagrams, syntax
highlighting, and slide layouts.

Install: `brew tap moravio/md-pdf && brew install md-pdf`
Requires: Node.js 18+, Chromium-based browser (e.g. Google Chrome).

---

## CLI usage

```
md-pdf <input.md> [output.pdf]
```

Arguments:
- `input.md`  — path to the Markdown source file (required)
- `output.pdf` — output path (optional, defaults to `<basename>.pdf` in cwd)

Options:
- `-h, --help`    — show help
- `-v, --version` — show version
- `--open`        — open the generated PDF after conversion
- `--watch`       — watch .md and .meta.json for changes, regenerate on save
- `--spellcheck`  — run spell check before PDF generation (language from .meta.json)
- `--page-format <fmt>` — override page size: `A3`, `A4`, `A5` (CLI > meta.json > branding default)

Other commands:
- `md-pdf-init <outDir> [--template=proposal] [--lang=cs|en|both]`
                                — scaffold a project from a bundled template.
                                  Default template `proposal` copies the Moravio
                                  CZ + EN proposal (with shared partials and assets)
                                  and uses the built-in `moravio-proposal` branding.
- `md-pdf-branding-init <dir>`  — scaffold a new branding folder
- `md-pdf-branding-check <dir>` — validate a branding folder

## Library API (since 3.12.0)

For Node.js consumers that want to embed md-pdf without spawning the CLI as a
subprocess (Electron apps, web servers, build pipelines):

```js
import { render } from "@moravio/md-pdf";          // or "@moravio/md-pdf/api"

const { outputPath, simple } = await render({
  mdPath: "/abs/path/document.md",
  outputPath: "/abs/path/document.pdf",  // optional; defaults to <basename>.pdf
  pageFormat: "A4",                       // optional; A3 | A4 | A5
  allowSimpleFallback: true,              // optional; default true
});
```

`render()` returns `{ outputPath, simple }`. `simple` is `true` when no
`.meta.json` was found and the no-frills path was used.

`MissingMetaError` is also exported. Set `allowSimpleFallback: false` to
make missing meta a hard error.

The CLI binary (`md-pdf`) and all its flags continue to work. Internally the
CLI calls `convertToPdf` / `convertToPdfSimple` from the same module, so the
library and CLI produce identical output.

Subpath exports:
- `@moravio/md-pdf` and `@moravio/md-pdf/api` — library entry (`render`)
- `@moravio/md-pdf/cli` — CLI entry (rarely needed; prefer the bin)

### HTML-only rendering (since 3.19.0)

Produce the same HTML + inlined CSS that PDF rendering would feed to
Puppeteer, but without launching Puppeteer. Use this when you need a
preview surface (Electron iframe, snapshot test, server-side render)
that visually matches the PDF without paying the Chromium startup
cost.

```js
import { renderToHtml } from "@moravio/md-pdf";

const result = await renderToHtml({
  mdPath: "/abs/path/document.md",
  mdContent: editorBuffer,                             // optional — render unsaved edits
  pageFormat: "A4",                                    // optional
  highlightTheme: "github-dark",                       // optional, default 'github'
  urlTransform: (abs) => `mdfont://${abs}`,            // optional
  baseHrefOverride: (baseDir) => `mdimg://_${baseDir}/`,
                                                       // optional — function or string
  allowMissingMeta: true,                              // optional, default true
  previewMode: true,                                   // optional, default false
  brandingOverride: "moravio-proposal",                // optional (since 3.20.0)
  brandingDirOverride: "/path/to/brand",               // optional (since 3.20.0)
});

// result.html         — complete <!DOCTYPE html>...</html> string
// result.inlinedCss   — concatenated CSS payload (already inside .html)
// result.assetBaseDir — directory of the source .md (or project root if synthesized)
// result.includedPaths — every {% include %} partial that contributed
// result.warnings     — non-fatal include-resolution issues
```

**`previewMode`** (since 3.19.0): switches rendering into a "paper sheet"
layout for screen preview. Body opts into `class="md-pdf-preview-mode"` and
`data-page-format="A3|A4|A5"`. An inline preview-only stylesheet:
- Sizes `.md-pdf-page` to the chosen format (1123 / 794 / 559 px at 96dpi),
  with paper shadow + sheet border.
- Splits content on `<div class="page-break"></div>` into one
  `.md-pdf-page` per source page (Adobe-Acrobat-style stacked sheets).
- Applies the brand's `contentMargins` as inner padding so the printable
  area matches the eventual PDF.
- Replaces unresolvable `{% include %}` directives with a styled
  `.md-pdf-include-error` rectangle (with `data-spec` and `data-reason`
  attributes) so broken includes are visible at a glance.

PDF flow leaves all preview-only chrome off.

**`baseHrefOverride`** accepts a string (verbatim) or a function
`(absoluteBaseDir) => string`. The function form is called with the resolved
asset baseDir AFTER any project-root walk-up the no-meta fallback may have
done — pass it to ensure relative `<img src>` URLs resolve to the right
directory.

**`allowMissingMeta`** (default true for `renderToHtml`): when the sibling
`.meta.json` is absent, walk up to find the project root and inherit
`branding` / `language` / `brandingDir` / `typography` from there. The
synthesized meta only overrides `title` to the partial's filename, so a
partial opened standalone in a preview gets the parent project's
brand-scoped CSS instead of `moravio-default`. Set `false` to require meta
and throw `MissingMetaError` (use this for snapshot tests / strict CI).

Skipped vs PDF flow (will land in subsequent versions if needed):
- Page-frame wrapping (the per-page top/right/bottom/left margin chrome).
- Watermark overlay (it's a position:fixed PNG; preview iframe is screen
  media, not paginated).
- Mermaid prerender — fenced ```mermaid blocks become
  `<div class="mermaid">…</div>` for the host to render client-side.
- TOC and title page (PDF needs page numbers; preview can't have them).

Properties:
- Never mutates `process.env` — safe to call concurrently with itself or
  alongside `render()`.
- No subprocess spawn, no Puppeteer launch.
- Throws `MissingMetaError` only when `allowMissingMeta` is false.

### Meta validation (since 3.15.0)

Before invoking the CLI or `render()`, callers can validate a parsed
`.meta.json` against the same rules the renderer enforces:

```js
import { validateMeta } from "@moravio/md-pdf";

const meta = JSON.parse(fs.readFileSync("doc.meta.json", "utf8"));
const { ok, errors, warnings } = validateMeta(meta);
if (!ok) {
  for (const e of errors) console.error(`${e.path}: ${e.message}`);
  process.exit(1);
}
```

Returns `{ ok: boolean, errors: ValidationIssue[], warnings: ValidationIssue[] }`,
each issue is `{ path: string, message: string }`. `path` is a JSON-pointer-ish
dotted key (`watermark.repeat.spacingX`, `customFields[2].label`). Errors block;
warnings (e.g. unknown top-level keys) never do — that keeps a desktop bundled
against an older md-pdf from rejecting a meta authored against a newer CLI.

The CLI runs the validator automatically before each conversion, so a typo
like `"pageFormat": "Letter"` aborts with a precise message instead of
silently rendering with the wrong paper.

Limitations: real-time progress streaming and `Buffer` returns are not yet
supported. Internal helpers (`build-toc`, `merge-pdf`, `prerender-mermaid`)
still run as `spawnSync(node, …, { stdio: "inherit" })`, so log lines go to
the host process's stdout. To capture them, hook `process.stdout.write`
while the render is in flight.

## File structure

Each document consists of:
1. `document.md` — the Markdown body content
2. `document.meta.json` — metadata file (same basename, optional)

When `.meta.json` is missing, md-pdf generates a simple PDF with no title page,
TOC, or pagination. When present, full branding and layout features are enabled.

## Metadata (.meta.json)

### Required fields

| Field      | Type   | Description                           |
|------------|--------|---------------------------------------|
| `title`    | string | Document title (shown on title page)  |
| `layout`   | string | `"document"` or `"slides"`            |
| `language` | string | `"en"` or `"cs"`                      |

### Optional fields

| Field          | Type    | Default            | Description                                                |
|----------------|---------|--------------------|------------------------------------------------------------|
| `subtitle`     | string  | —                  | Subtitle text                                              |
| `author`       | string  | —                  | Author name                                                |
| `client`       | string  | —                  | Client or organization                                     |
| `email`        | string  | —                  | Contact email (shown on title page as mailto link)         |
| `phoneNumber`  | string  | —                  | Contact phone (shown on title page as tel link)            |
| `date`         | string  | today's date       | Explicit date string; localized by `language` when omitted |
| `hideDate`     | boolean | `false`            | Drop the Date row from the document title meta grid and the Date column from the slides title footer. For storybook covers where a printed date is out of place. |
| `customFields` | object  | —                  | Label → value pairs rendered below client on title page    |
| `titlePage`    | boolean | `true`             | Set `false` to skip the title page entirely                |
| `blankPageAfterTitle` | boolean | `false`     | Insert an unnumbered blank leaf after the title (booklet duplex printing). Pagination, when on, restarts at `1` on the first content page; total excludes title + blank. Ignored when `titlePage: false`. |
| `pagination`   | boolean | `true`             | Page numbers and auto slide breaks                         |
| `toc`          | boolean / int / object | `true` | Table of contents (document layout only). Accepts `true`/`false`, an integer 1–6 (shorthand for `{minHeadingLevel: 1, maxHeadingLevel: N}`), or `{minHeadingLevel?, maxHeadingLevel?}` (each in 1–6, min ≤ max). Default range is H1–H3. Use `{minHeadingLevel: 2}` to skip H1 if you treat it as a title. |
| `branding`     | string  | `"moravio-default"`| Built-in branding id                                       |
| `brandingDir`  | string  | —                  | Path to external branding folder (overrides `branding`)    |
| `pageFormat`   | string  | `"A4"`             | Page size: `A3`, `A4`, `A5`. Single-valued for the whole document — title page always matches body format. CLI `--page-format` wins. |

### Branding-level options (set in `branding.json`, not in `.meta.json`)

| Field                 | Type   | Description                                                                                                                                                                       |
|-----------------------|--------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `pageBackground`      | string | CSS color (`"#fbf6ed"`, `"rgb(...)"`). Filled on every printed sheet via fixed-position body::before so multi-page overflow stays opaque. Use when the brand has a non-white page. Title page is full-bleed too — title pdf margin is moved to body padding when set. |
| `titlePage.<layout>.decorationInside` | boolean | When `true`, the brand decoration is placed inside the title container/outer wrapper. Use for tall portrait illustrations that would otherwise overflow to a second sheet. |
| `footer.fontRegular`  | string | Path to a TTF (relative to branding.json). Used to embed the page-number footer instead of Helvetica.                                                                              |
| `toc.fontRegular` / `toc.fontBold` | string | Path to a TTF (relative to branding.json). Embed in the rendered TOC overlay so chapter/page text matches the brand body font. |
| `watermark`    | object  | —                  | Text watermark config (or `false` to disable brand default)|
| `dictionary`   | array   | —                  | Extra words for `--spellcheck`                             |

## Layout: document vs slides

### Document layout (`"layout": "document"`)
- A4 portrait
- Title page → Table of contents → Body content
- Page numbers and footer logo on every page (except title page)
- TOC auto-generated from `#`, `##`, `###` headings by default. Disable with
  `"toc": false`; widen with `"toc": 4` (shorthand for H1–H4) or
  `"toc": {"minHeadingLevel": 2, "maxHeadingLevel": 4}` for an explicit range
- Headings converted to clickable anchors in PDF
- `#` lines inside fenced code blocks (``` or `~~~`, max 3-space indent),
  HTML comments (`<!-- ... -->`), and raw-text HTML blocks (`<script>`,
  `<pre>`, `<style>`, `<textarea>`) are NOT extracted — shell comments,
  preprocessor directives, etc. stay out of the TOC

### Slides layout (`"layout": "slides"`)
- A4 landscape, larger typography
- Title page → Slide content (no TOC)
- With `"pagination": true` (default):
  - Automatic page breaks before each boundary heading (`#` or `##`)
  - If file has `#` headings, those are boundaries; otherwise `##` is used
  - `###` headings never trigger automatic breaks
  - `#` lines inside code fences, HTML comments, or `<script>`/`<pre>`/
    `<style>`/`<textarea>` blocks never trigger breaks either
  - Manual `<div class="page-break"></div>` blocks coexist with auto breaks:
    auto-injection still fires at heading boundaries where no manual break exists
- With `"pagination": false`:
  - No automatic breaks; only manual `<div class="page-break"></div>` works
  - No page footer (no "n / total" strip)
  - Slightly smaller bottom margin

## Markdown syntax

Standard GitHub Flavored Markdown (GFM) via marked 9.x:
- **Bold**, *italic*, ~~strikethrough~~, `inline code`
- Tables, ordered/unordered lists, blockquotes
- Fenced code blocks with syntax highlighting (highlight.js 11.x)
- Images (relative paths resolved from .md file directory)
- Links, horizontal rules (hidden in print CSS)

### YAML front matter

```markdown
---
title: Internal note
---
```

Front matter is **stripped** before processing — it never appears in the PDF.
Use `.meta.json` for all metadata, not front matter.

### Includes (md-pdf 3.17+)

Inline a partial Markdown file with `{% include path %}` on its own line:

```markdown
# Proposal

{% include partials/about-moravio.cs.md %}

## Project specifics
…
{% include @moravio/contact-footer.cs %}
```

Resolution rules:
- Relative paths (`partials/foo.md`) resolve against the host file's directory.
- Absolute paths are used as-is.
- `@<brand>/<name>` resolves to `~/.md-pdf/brands/<brand>/partials/<name>[.md]`,
  falling back to bundled `brandings/<brand>/partials/`. Override the lookup
  root with `MD_PDF_BRANDS_DIR`.
- `@bundled/<rel>` (since 3.20.0) resolves against the md-pdf package install
  dir — cross-platform reference to shipped examples/partials without an
  absolute path. Example: `@bundled/examples/include/partials/o-moraviu.cs.md`.

The directive must be the only content on its line (leading/trailing
whitespace ignored); inline forms in prose or fenced code blocks are
not expanded. Cycles are detected (warning by default, throw in strict
mode via the library API); default depth limit is 8. Missing files
emit a warning and leave the literal directive in the PDF.

`md-pdf --watch` auto-watches every transitively-included file —
editing a partial fires a re-render.

## Page breaks

Insert a page break anywhere using:

```html
<div class="page-break"></div>
```

This works in both document and slides layouts.

In slides with `pagination: true`: manual page breaks and automatic heading
breaks coexist. You can place manual breaks mid-slide to split long content,
and the tool still auto-injects breaks at heading boundaries.

## Mermaid diagrams

Fenced `mermaid` blocks are pre-rendered to static SVG before PDF generation.
All Mermaid diagram types are supported (Mermaid 11.x).

````markdown
```mermaid
flowchart LR
  A[Input] --> B[Process] --> C[Output]
```
````

### Diagram size control

Wrap the mermaid block in a `<div>` with size classes:

Height classes (millimeters):
- `mermaid-h20`, `mermaid-h40`, `mermaid-h60`, `mermaid-h80`
- `mermaid-h100`, `mermaid-h120`, `mermaid-h160`, `mermaid-h200`
- `mermaid-h-auto` (no limit)

Width classes (percentage of content area):
- `mermaid-w25`, `mermaid-w33`, `mermaid-w50`
- `mermaid-w66`, `mermaid-w75`, `mermaid-w100`

Example:
````markdown
<div class="mermaid-h80 mermaid-w50">

```mermaid
flowchart TD
    A --> B --> C
```

</div>
````

### Per-diagram configuration

Use YAML frontmatter inside the mermaid block:

````markdown
```mermaid
---
config:
  gantt:
    fontSize: 14
    barHeight: 28
---
gantt
    title Timeline
    dateFormat YYYY-MM-DD
    section Phase 1
    Task A :a1, 2026-01-01, 30d
```
````

Gantt month labels are automatically localized based on `language` in .meta.json.

## Custom branding

### Using external branding (recommended for consumer projects)

```bash
# 1. Scaffold
md-pdf-branding-init ./my-brand

# 2. Edit files:
#    my-brand/branding.json    — colors, fonts, margins
#    my-brand/logo.svg         — header logo
#    my-brand/decoration.svg   — title page decoration
#    my-brand/footer-mark.svg  — footer icon

# 3. Validate
md-pdf-branding-check ./my-brand

# 4. Reference in .meta.json (path relative to .meta.json location):
#    "brandingDir": "./my-brand"
```

### User-installed brandings (shared across projects)

For brandings you reuse across many projects, install them under
`~/.md-pdf/brands/<id>/` (cross-platform — resolved via `os.homedir()`).
Reference them from any `.meta.json` by id, exactly like a built-in:

```jsonc
{ "branding": "my-brand" } // resolves ~/.md-pdf/brands/my-brand/
```

Resolution order for `branding: "<id>"`:

1. `~/.md-pdf/brands/<id>/` (user-installed, takes precedence — lets you
   shadow a built-in like `moravio-default` without forking)
2. Bundled `<package>/brandings/<id>/` (fallback)

Override the user dir with the `MD_PDF_BRANDS_DIR` env var (useful for
tests, CI, or project-scoped overrides). Each user brand folder must
contain a `branding.json`; scaffold one with
`md-pdf-branding-init ~/.md-pdf/brands/my-brand`.

### branding.json structure (version 1)

| Section          | Purpose                                                        |
|------------------|----------------------------------------------------------------|
| `version`, `id`  | Must be `1` and match the folder name                         |
| `assets`         | `logo`, `decoration`, `footerMark` — SVG paths                |
| `fonts`          | `sansStack`, `monoStack`, `fontCssDocument`, `fontCssSlides`   |
| `contentMargins` | `document`/`slides` with `paginationOn`/`paginationOff`        |
| `contentCss`     | CSS variables for colors, sizes (camelCase → --brand-kebab)    |
| `footer`         | Page footer settings (margins, font size, mark height)         |
| `toc`            | Table of contents styling (margins, colors, typography)        |
| `titlePage`      | `document`/`slides` layout with inline style strings           |
| `watermark`      | Optional brand-wide default watermark (overridable per-doc)    |

The title page loads the generated `brand-vars.css` (same as content pages),
so `@font-face` rules from `fontCssDocument` / `fontCssSlides` apply there
too. Reference the branded family directly in `titlePage.*Style` (e.g.
`font-family: 'Montserrat', sans-serif`) to keep typography consistent
across title page, TOC, and body.

## Watermark

Stamp a repeated or single text watermark behind the content (title page,
TOC, and body). Set in `.meta.json` (per-document) or in `branding.json`
(brand-wide default). `.meta.json` overrides the brand default; use
`"watermark": false` (or `null`) in `.meta.json` to disable a brand default.

```json
{
  "watermark": {
    "text": "DRAFT",
    "font": "Montserrat, sans-serif",
    "size": "72pt",
    "color": "rgba(198, 40, 40, 0.18)",
    "weight": 800,
    "angle": -30,
    "position": "center",
    "repeat": { "spacingX": "9cm", "spacingY": "7cm" }
  }
}
```

| Field      | Type                       | Default                                 | Description                                                  |
|------------|----------------------------|-----------------------------------------|--------------------------------------------------------------|
| `text`     | string (required)          | —                                       | The label to render                                          |
| `font`     | string (CSS family)        | brand `fonts.sansStack` (→ `sans-serif`)| Any CSS `font-family` value. The SVG runs inside a `data:` URI — `var()` refs do NOT cascade in, so pass a concrete stack; the default is inlined at build time |
| `size`     | string (CSS length)        | `56pt`                                  | Any CSS size (`pt`, `mm`, `px`, …)                           |
| `color`    | string (CSS color)         | `rgba(0, 0, 0, 0.055)`                  | Hex/rgba; include alpha (default is faint — tune per brand)  |
| `weight`   | number 1–1000 or CSS kw    | `600`                                   | `400`, `600`, `700`, `bold`, `normal`, …                     |
| `angle`    | number (degrees)           | `-30` if `repeat`, else `0`             | Rotation                                                     |
| `position` | preset string or `{x, y}`  | `center`                                | Presets: `center`, `top`, `bottom`, `left`, `right`, `top-left`, `top-right`, `bottom-left`, `bottom-right`. `{x, y}` accepts any CSS length/percentage |
| `repeat`   | `{ spacingX, spacingY }`   | —                                       | Tiles the watermark at this step. Omit for single-per-page   |
| `scope`    | `"all"` \| `"title"` \| `"content"` | `"all"`                                 | Restrict the watermark to the title page only, content pages only, or both (default). Use `"title"` for a "DRAFT" stamp on the cover that doesn't bleed into the body. |

**Strict validation:** unknown keys in `watermark`, `repeat`, or `position`
throw an error at conversion time.

Implementation detail: the watermark is a `body::before` background painted
with `position: fixed; inset: 0`, which Chromium replicates on every printed
page across the full paper. For repeat watermarks with absolute-unit spacing
(mm/cm/in/pt/pc/px) the SVG is pre-rasterized to a PNG via Puppeteer at
300 DPI, so Chromium emits the watermark as a single PDF Image XObject reused
across pages — both pinning the perceived intensity at any zoom level (no
per-page rasterization variance) and keeping PDF size roughly constant on
long documents instead of scaling with page count. Single (non-repeat)
watermarks and tiles with relative units (vw/vh/em/%) keep the SVG path.
Content and TOC renders always run with `pdf_options.margin: "0"` and derive
their per-page margins from a paged-media frame table
(`<table class="mp-page-frame">` with `<thead>`/`<tfoot>` spacers) built from
the brand's `contentMargins`.
The title render still uses `pdf_options.margin: "0"` + `body { padding }`
when the watermark is active (single-page layout where body padding works).
All built-in brandings set `printBackground: true` in their title-page
`pdfOptions`, which is required for background images to print.

**Brand-author constraint (external `brandingDir`):** any brand CSS that
targets table cells (`td`, `th`, `thead`, `tbody`, `tfoot`) must exclude the
frame with `:not(.mp-page-frame)`. Otherwise cell padding / borders / zebra
rules leak into the per-page spacers and distort margins. The built-in
brandings already scope their table rules this way; audit external `branding.json`
CSS once before enabling the watermark on that brand.

**Title-page layout note.** The `pdf_options.margin: 0` switch enlarges
the Chromium viewport, shifting `position: fixed` brand decoration by the
original margin. Brands can restore the original visual spot with an
optional per-layout field:

```jsonc
"titlePage": {
  "document": {
    "watermarkDecorationTransform": "translate(0, 20mm)"
  }
}
```

Emitted verbatim as `.brand-title-decoration { transform: <value> }` in the
generated watermark CSS (the title `<img>` decoration is tagged with that
class). Tune empirically per brand — different decoration reference sides
(top/right/bottom/left) mean no single formula fits all brands. Leave unset
to skip the override.

## Typography normalization

Since v3.7.0 md-pdf runs the Markdown source through a small typography
rule table before rendering. Rules target characters that AI assistants
emit but that are unwanted in client-facing documents.

**Current rules:**

| Rule id   | Replaces                                  | With              |
|-----------|-------------------------------------------|-------------------|
| `em-dash` | `—` (U+2014) + any surrounding ` `/`\t`   | ` - ` (spaced hyphen) |

Rule `em-dash` collapses `word—word`, `word — word`, `word —word`,
`word— word` all to `word - word`. Newlines are NOT consumed — em-dash on
its own line produces ` - ` on its own line.

**Skip zones** (rules NEVER apply, contents preserved verbatim):

- Fenced code blocks (```` ``` ````)
- Inline code (`` ` ``)
- HTML `<pre>` and `<code>` blocks
- Markdown link/image targets: the `(url)` part of `[text](url)` and `![alt](url)`
- Bare URLs (`https?://…`)

**Opt-out via `.meta.json`:**

```json
{
  "typography": {
    "preserve": true
  }
}
```

Or granularly:

```json
{
  "typography": {
    "disable": ["em-dash"]
  }
}
```

| Field      | Type              | Default | Description                                  |
|------------|-------------------|---------|----------------------------------------------|
| `preserve` | boolean           | `false` | `true` skips ALL typography rules            |
| `disable`  | string[] (ids)    | `[]`    | Disable specific rules by id                 |

Invalid shapes (non-object root, non-boolean `preserve`, non-string-array
`disable`) are warned about and ignored (graceful — normalization is a
polish feature, not a blocker). Unknown rule ids in `disable` are warned
about and skipped.

**When to opt out:** rarely. Em-dash normalization is desirable for
client-facing output. Opt out only when the em-dash carries semantic
meaning that must be preserved (academic typography, quoted source text,
etc.). Prefer `disable: ["em-dash"]` over `preserve: true` so future
rules still apply.

**Examples:**

- `examples/document/typography-test.md` — normalization on (default).
- `examples/document/typography-preserve-test.md` — `preserve: true`.

Implementation: `lib/typography.js` exports `TYPOGRAPHY_RULES` (the rule
table) and `applyTypographyRulesToMarkdown`. The table is the single
source of truth; adding a new rule is one entry `{ id, pattern, replacement }`
(the `pattern` RegExp must have the `g` flag). Skip logic is applied
uniformly to every rule.

## Common tasks — quick reference

### Generate a simple PDF (no metadata)
```bash
md-pdf notes.md
```

### Generate a branded document
```bash
md-pdf report.md --open
```
(requires `report.meta.json` next to `report.md`)

### Watch mode for iterating
```bash
md-pdf report.md --watch --open
```

### Slides with auto page breaks
Create `slides.meta.json`:
```json
{
  "title": "Team Update",
  "layout": "slides",
  "language": "en"
}
```
Write `slides.md` with `##` headings as slide boundaries:
```markdown
## Agenda

- Item 1
- Item 2

## Progress

We completed X, Y, Z.

## Next steps

Action items here.
```
Each `##` becomes a new slide automatically.

### Slides with mixed manual + auto breaks
Use `<div class="page-break"></div>` to split a slide mid-content.
Auto breaks still fire at heading boundaries:
```markdown
## Architecture

Table and text here...

<div class="page-break"></div>

Diagram on its own page.

## Summary

Wrap-up slide (auto break before this ##).
```

### Document without title page
```json
{
  "title": "Appendix",
  "layout": "document",
  "language": "en",
  "titlePage": false
}
```

### Document without TOC
```json
{
  "title": "Short Brief",
  "layout": "document",
  "language": "en",
  "toc": false
}
```

### Czech document


## Troubleshooting

| Problem | Solution |
|---------|----------|
| "Metadata file not found" | Create a `.meta.json` with same basename as `.md`, or let md-pdf generate a simple PDF |
| Mermaid diagram missing | Check diagram syntax; errors print to console |
| PDF looks different locally vs CI | Ensure same Chrome/Chromium version; use local fonts (`fontCss…`) |
| Spell check false positives | Add words to `"dictionary"` array in `.meta.json` |
| Custom brand not working | Run `md-pdf-branding-check ./path` to validate |
| Page break not working | Use exactly `<div class="page-break"></div>` on its own line |
| Slides have no page numbers | Set `"pagination": true` in `.meta.json` |
